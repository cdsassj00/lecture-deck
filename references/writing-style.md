# 강의안 문체 규칙 (반드시 지킬 것)

이 규칙들은 실제 임원 강의안 제작에서 여러 번 반려당하며 정리된 것이다. 어기면 결과가 거부된다.

## 1. 제목 — 건조한 기능적 명칭

`[구분] 주제 — 서술형 부제` 형식. 한 소주제 묶음에 걸쳐 같은 제목을 반복해도 된다(교안의 챕터 헤딩처럼).

**금지 (비유·상징·수사·슬로건·블로그체):**
- ✗ "나의 첫 하네스 — 헌법과 스킬을 얹기"
- ✗ "잘 만든 하네스는 어떻게 생겼나"
- ✗ "두 가지가 하나의 이름을 함께 쓰고 있다"
- ✗ "이 말은 성립하는가" (수사적 질문)
- ✗ "프롬프트를 쌓지 마라, 흐름을 설계하라" (슬로건/대구)

**권장 (사실·기능 중심):**
- ✓ "[실습 ①] 지침과 스킬 적용 — 적용 전후 비교"
- ✓ "[사례] 황민호 하네스 — 4계층 문서 구조"
- ✓ "[이론] 좁은 의미와 넓은 의미"
- ✓ "강의 요약" (← "오늘 우리가 함께 본 것" 같은 표현 금지)

## 2. 본문 — 합니다체 설명문

- 개념을 **합니다체 설명 문단**으로 푼다. 예: "우리가 쓰는 OO는 사실 …하는 구조입니다."
- 불릿 나열보다 **번호 개념 정의**(이름 + 2~3줄 설명)를 쓴다.
- 요약 슬라이드는 **사실 진술체**(한다체)로: "OO는 두 가지를 가리킨다. …" — 슬로건 금지.

## 3. 비개발자 눈높이

대상이 비개발자(임원·일반인)일 때:
- 전문용어는 **풀어서** 쓰고 괄호로 원어 병기. 가능하면 "예 — …" 한 줄로 구체화.
- 추상적 동사보다 **구체적 동작**: "전송/정지 판정/회신" → "묻기 / 고르기 / 확인 후 실행 / 알려주기".
- 개발 용어 **금지**: PR·머지·린터·CI·커밋·레포 → 평이한 말로 치환.
  - 린터/CI/검사 → "어기면 멈추는 자동 검사 장치"
  - 코드가 막는 구조 → "통과 못 하면 다음 단계가 안 열리는 관문"
- 비유에 기대지 말 것(이 사용자는 비유 표현을 특히 싫어함). "헌법" 같은 비유어도 "기준 문서"로.

## 4. 메타 문장 금지

수강생이 보는 본문만 쓴다. "강사는~", "수강생은~", "이 강의는 ~방식으로 운영" 같은 **제작자 시점 문장 금지**. 청중 직접 지시가 꼭 필요할 때만 "여러분".

## 5. 폰트 크기

본문이 너무 작으면 반려된다. 1280×720 캔버스에서 **설명문 16.5px, 개념·표·단계 14~15px**(≈10~11pt 체감)을 기준으로. 템플릿 CSS가 이미 이 크기이니 줄이지 말 것.

## 6. 출처·수치

- 사례·통계는 출처를 `.srcline`에 단다.
- 수치는 `.stat`으로 강조(예: 49.5→79.3, +60%).
